# Week 2 — Competency 2: Code literacy and documentation

Personal notes on what this competency means in practice for this class and how this week’s project connects to it.

## What Competency 2 means to me

**Code literacy** is the ability to **read a block of code and explain what it does**: what runs when, what data it touches, and what it produces. It is less about memorizing syntax and more about following logic step by step.

**Documentation** should help someone with an HCD background (not necessarily a software engineer) in all of these ways:

- **Purpose** — why the script or artifact exists
- **Inputs and outputs** — what goes in (e.g., files, columns) and what comes out (e.g., printed summary, rows on a page)
- **How to run it** — enough for a classmate to reproduce results
- **Where data lives** — filenames, locations, and how the code finds them

## What this week’s project shows

The work that best reflects my **code literacy** so far is **handling the CSV of app reviews**: a structured table with consistent columns, read in Python and used to compute word counts per review.

The file I use as the clearest **documentation + code** example is **`demo_word_count.py`** — it ties the data file, the processing steps, and the printed results together in one place.

## What was hardest

The hardest part of the assignment workflow was **committing changes and getting them to GitHub** (version control and remote repo steps), not the CSV or Python logic itself.

## What improved

Through this week I got more comfortable with:

- **Structuring and maintaining data in CSV** (headers, rows, review text as a stable column)
- **Using Cursor** as a day-to-day environment for editing and collaborating on files

## Optional follow-up

If the course asks for a growth goal later, a natural next step would be to deepen **Git** confidence (commits, remotes, and knowing what is tracked) so it matches my growing comfort with data and small scripts.
